SPARK MANUAL v2.0

Write once. Run anywhere. Own your tools.

Spark is a mobile-first language that compiles to standalone HTML. Write once, share a file, runs anywhere. No server, no install, no build step.

Keywords are short (5 chars max) — easier to type on a phone. Programs split into named Windows (like scenes or classes). MAIN is always the entry point.

Design rule: Spark source code never uses parentheses, brackets, or braces. Everything is keyword + space-delimited arguments. Simple to type, impossible to get wrong.

LINE FORMAT

Every line starts with a number. Numbers determine execution order, not physical position.

10 asign SCORE = 0
20 asign LIVES = 3
30 goto 100
100 print SCORE

Space in increments of 10 so you can insert lines later. Reference lines in another window with NUM-WINNAME:

30 goto 10-GAMEOVER     rem jump to line 10 in GAMEOVER window
40 calsb 100-ENEMY      rem call sub at line 100 in ENEMY window

VARIABLES

Variables always have a window suffix. Inside the current window the suffix is inferred automatically.

10 asign SCORE = 0         rem SCORE-MAIN in MAIN window
20 asign HP-ENEMY = 100    rem explicit cross-window variable
30 print SCORE             rem reads SCORE-MAIN
40 print HP-ENEMY          rem reads from ENEMY window

Variables hold numbers or strings. Math on numeric strings works. String concatenation uses +. Strings can be compared in IF conditions with = and <>.

10 asign X = "5"
20 asign Y = X + 3         rem Y = 8

10 asign NAME = "DOOM"
20 asign VER = 1
30 print NAME + " v" + VER    rem prints: DOOM v1

10 IF NAME = "JON" THEN PRINT "welcome"
20 IF STATUS <> "done" THEN GOTO 10

CONTROL FLOW

IF / THEN / ELSE

10 IF HP < 1 THEN CALSB DEAD
20 IF X > W-GRAPH AND Y > H-GRAPH THEN PRINT "out of bounds"
30 IF SCORE > 100 OR LIVES = 0 THEN ASIGN DONE = 1
40 IF HP > 0 THEN PRINT "alive" ELSE PRINT "dead"
50 IF X < 0 THEN GOTO 100 ELSE ASIGN X = X + 1
Supports AND, OR, NOT in conditions. Also <> for not-equal. ELSE is optional — provides an alternate branch on the same line.

MOD Operator

Modulo (remainder) can be used directly in expressions:

10 ASIGN R = X MOD 3
20 IF COUNT MOD 10 = 0 THEN PRINT "milestone"

WHILE / ENDWH REMOVED

WHILE/ENDWH has been removed from the language. Use GOTO for backward loops or STRLP/NDLP for counted loops instead.

Colon Chaining

Multiple statements on one line, separated by : (space-colon-space):

10 ASIGN X = 0 : ASIGN Y = 0 : ASIGN VX = 4
20 CALSB CLEAR-GRAPH : CALSB COLOR-GRAPH : CALSB CIRC-GRAPH
Each chained statement is expanded to its own line internally. All keywords except STRLP/NDLP work in chains.

STRLP / NDLP (for loop)

10 strlp I = 1 to 8
20   strlp J = 1 to 8
30     rem draw grid at I,J
40   ndlp
50 ndlp

rem with step:
10 strlp X = 0 to W-GRAPH step 40
20   rem draw vertical line every 40px
30 ndlp

GOTO game loop

80 wait 16
90 calsb TICK-MAIN
100 goto 80             rem loops forever

10 calsb 200            rem jump to 200, return after rturn
20 calsb SQRT-MATH      rem call package function
200 asign Y = Y + 1

Inline functions usable anywhere in ASIGN, IF, or PRINT expressions. Arguments are space-delimited — Spark uses no parentheses, brackets, or braces anywhere in source code:

LOCAL variables

Declare variables local to a sub window. Without LOCAL, all variables are global.

CALC window:
10 LOCAL X, Y, TEMP
20 ASIGN X = 5
30 ASIGN Y = 10
40 ASIGN RESULT = X + Y
50 RTURN
LOCAL must be the first statement in the window. Variables are let-scoped inside the compiled sub function — they don't pollute or read from global state.

ONERR — error handling

Place ONERR anywhere in your program. All subsequent CALSB calls get wrapped in try/catch. Errors are captured in the _sparkERR variable instead of crashing.

10 ONERR
20 CALSB SOMETHING-RISKY
30 IF _sparkERR <> "" THEN PRINT "Error: " + _sparkERR
40 END
ONERR is a program-wide flag — once set, ALL CALSB calls in the program are wrapped. Check _sparkERR after any call that might fail.

WINDOWS

Windows are separate code spaces. MAIN is the entry point. Each window has its own line numbers and variable namespace. GOWIN switches one-way. CALSB with cross-window address calls and returns.

rem -- MAIN
10 imprt store
20 calsb GET-STORE
30 gowin INTRO

rem -- INTRO
10 imprt graph
20 imprt input
30 calsb INIT-INPUT
40 if TAPD-INPUT = 1 then gowin GAME
50 calsb POLL-INPUT
60 wait 16
70 goto 40

PACKAGES

Import a package, set its input variables, call a function, read the result. All package vars use -PKGNAME suffix.

LOCAL VARIABLES

Prefix any variable name with $ to make it local to the current subroutine. Locals are automatically deleted when the subroutine returns — they never leak to the caller or other subroutines. Works with both asign and bare assignment.

rem calling code
10 asign X = 100
20 calsb DOUBLE
30 print X          rem still 100 - X was not touched
40 print $RESULT    rem error - $RESULT is gone after rturn
50 print RESULT-MAIN rem use a named return var instead
60 end

100 calsb DOUBLE
110   asign $N = X * 2   rem $N is local
120   asign RESULT-MAIN = $N
130 rturn

Recursive example — factorial

10 asign N = 6
20 calsb FACT
30 print RESULT-MAIN
40 end

100 rem FACT: computes N! into RESULT-MAIN
110 asign $N = N
120 if $N <= 1 then asign RESULT-MAIN = 1
121 rturn
130 asign N = $N - 1
140 calsb FACT
150 asign RESULT-MAIN = $N * RESULT-MAIN
160 rturn
$ locals are scoped to call depth — each nested CALSB gets its own frame. On RTURN all $vars at that depth are wiped. Return values should use named package-style vars (e.g. RESULT-MYNAME) since those persist across calls.

Gotchas

Never put " inside an HTML-DOM or TXT-DOM value — it ends the Spark string and causes parse errors. Use CSS classes (injected via a style element) and unquoted HTML attributes instead.

FLOW-DOM = 1 sets position:relative so elements stack vertically — useful for items inside a PAGE container. FLOW-DOM = 0 (default) uses absolute pixel coordinates.

GOTCHAS

Colon chaining spacing matters

Colon chaining requires exactly : (space-colon-space). Missing spaces will cause errors or silent failures.

rem WRONG — no spaces around colon:
10 asign X = 1:asign Y = 2

rem RIGHT — space-colon-space:
10 asign X = 1 : asign Y = 2

IF THEN RTURN — interpreter only gotcha

In interpreter mode, IF cond THEN RTURN silently fails. Use GOTO + RTURN instead. In compiled mode (RUN button), this works fine.

rem SAFE for both modes:
100 if X = 0 then goto 150
110 goto 200
150 rturn
200 rem ...rest of sub...

IF THEN GOTO with fall-through

When using IF cond THEN GOTO label, the next line runs if the condition is false. Use ELSE for a cleaner alternative, or add a skip-GOTO after conditional jumps.

rem CLEAN — using ELSE:
100 if DONE = 1 then goto 500 else goto 200
500 rturn
200 rem ...loop body...

rem ALSO WORKS — explicit skip:
100 if DONE = 1 then goto 500
110 goto 200
500 rturn
200 rem ...loop body...

Variables need exactly one hyphen (interpreter mode)

In interpreter mode, all variable names must contain exactly one hyphen. Plain names like X may not work, and names with two or more hyphens like IC-LEFT-RAW silently evaluate to 0. In compiled mode (RUN button), plain names work fine.

rem WRONG — no hyphen:
10 asign X = 5

rem WRONG — two hyphens, evaluates to 0:
10 asign IC-LEFT-RAW = 7

rem RIGHT — exactly one hyphen:
10 asign X-V = 5
20 asign IC-LRAW = 7

Line numbers sort numerically

Lines execute in numeric order regardless of where you write them. 8005 sorts after 829 — it does NOT go between 800 and 810. Space lines by 50s to leave room for insertions.

rem These execute as: 50, 100, 150, 200 — not in written order
200 print "last"
50  print "first"
150 print "third"
100 print "second"

BOOTSTRAP / SELF-HOSTING

Bootstrap means writing Spark in Spark. The goal: an interpreter for Spark, written entirely in Spark itself. This is how programming languages prove they work — when a language can run its own source code, it has reached self-hosting.

In practice this means writing a tokenizer, line parser, variable store, statement executor, and main loop all in Spark — using only the primitives Spark itself provides (STRING package, ASIGN, IF, GOTO, CALSB). The result is a second Spark interpreter living inside the first one, able to run Spark programs as data.

The multi-window architecture is used here deliberately — each component of the interpreter lives in its own window with clean line numbers, and they call each other with CALSB 100-WINDOWNAME.

Why it matters

Self-hosting is the foundation for a compiler. Once Spark can interpret Spark, the next step is to make it compile Spark — transforming Spark source into faster native Spark code or JavaScript. The bootstrap examples in the EX menu show each stage of this process.

Architecture

WINDOWROLE
MAINLoads the program to interpret into ML-PROG and calls LOOP
LOOPMain fetch/decode/execute cycle — gets each line, calls LPARSE then EXEC
EXECStatement executor — handles ASIGN, PRINT, END, GOTO, CALSB, RTURN, IF
VSTOREVariable store — SET at line 100, GET at line 400
LPARSELine parser — splits a line string into LP-NUM, LP-KEY, LP-REST

The interpreted program

Programs run by the bootstrap interpreter are stored as pipe-delimited strings in ML-PROG. Each segment is one line. The loop fetches them by index, LPARSE splits them, EXEC runs them.

10 imprt string
20 asign ML-PROG = "10 asign X-V = 3|20 print X-V|30 end"
30 calsb 100-LOOP
40 end

Calling across windows

rem LOOP calls LPARSE to parse a fetched line:
280 calsb 100-LPARSE

rem EXEC calls VSTORE to read a variable:
370 calsb 400-VSTORE

rem EXEC calls VSTORE to write a variable:
377 calsb 100-VSTORE

rem EXEC calls itself for GOTO/CALSB line search:
610 calsb 900-EXEC

Spark can edit Spark

Beyond self-hosting, Spark programs can directly read and modify the Spark editor itself at runtime. This is a separate capability — not just interpreting Spark as data, but reaching into the live editor and changing code, windows, and state.

This is possible through two packages:

  • UPDATE — fetch a new version of Spark from a URL and hot-swap it into the running editor. A Spark program can update the language it is running in.
  • SYS — read and write the contents of editor windows at runtime. A program can inspect its own source code, generate new code and write it into a window, or modify another window's program while running.

Combined with the bootstrap interpreter, this means a Spark program can: generate Spark source code, write it into a window using SYS, and then run it — a full code generation and execution pipeline entirely within Spark.

rem generate code and write it into a window at runtime:
10 imprt sys
20 asign CODE-SYS = "10 print hello from generated code"
30 asign WIN-SYS = GENERATED
40 calsb SET-SYS
rem now run it:
50 calsb 10-GENERATED

Current status

The bootstrap interpreter currently supports: ASIGN (with + expressions), PRINT, END, GOTO, CALSB, RTURN, and IF with all 6 comparison operators. Missing: -, *, / in expressions, WHILE loops, and the IMPRT bridge (letting interpreted programs call packages). See the TODO for the roadmap.

BOOTSTRAP INTERPRETER

Spark includes a self-hosted interpreter written entirely in Spark. Load the Bootstrap: Multi-Window example to access it. The interpreter runs Spark programs stored as pipe-delimited strings in the ML-PROG variable.

Architecture

WINDOWPURPOSE
MAINEntry point. Sets ML-PROG and starts the interpreter loop.
LPARSELine parser. Input: LP-IN (one line). Output: LP-NUM, LP-KEY, LP-REST.
VSTOREVariable store. Read/write interpreted program variables by name.
EXECStatement executor. Dispatches on LP-KEY to handle each keyword.
LOOPMain interpreter loop. Walks ML-PROG line by line, calls LPARSE+EXEC.
BRIDGEPackage bridge. Routes CALSB calls from interpreted programs to STRING/MATH/CONVERT.

ML-PROG format

Programs are stored as pipe-delimited lines in a single string:

ASIGN ML-PROG = "10 PRINT hello|20 ASIGN X = 5|30 PRINT X|40 END"

Each segment is one numbered Spark line. The interpreter parses and executes them in order, respecting GOTO, CALSB, RTURN, IF, and WHILE.

Supported keywords

ASIGN, PRINT, END, GOTO, IF/THEN, CALSB, RTURN, IMPRT — full expression evaluation including +, -, *, / with correct precedence.

Package bridge

Interpreted programs can call STRING, MATH, and CONVERT packages via the BRIDGE window. Set SE-BFUNC to the function name and SE-BINPUT/SE-BSTART/SE-BN/SE-BWITH to parameters, then CALSB 100-BRIDGE. Results come back in SE-BRSTR, SE-BRMTH, SE-BRCNV.

In practice, just use IMPRT and CALSB normally — EXEC intercepts package calls and routes them automatically.

BOOTSTRAP COMPILER

Three Bootstrap windows that compile Spark source to JavaScript — written entirely in Spark itself. This is the proof-of-concept self-hosting compiler. The same logic is also implemented natively in Spark's compileSpark() function, which powers CALSB COMPILE-Spark, ⬇ APP, and ⬇ SELF.

Windows

WINDOWPURPOSE
XFRMExpression transformer. Wraps hyphenated variable names in v['...'] for JS output.
CGENCode generator. Takes one parsed line (CG-NUM, CG-KEY, CG-REST) and emits one JS case N: statement.
COMPCompiler loop. Walks ML-PROG, calls LPARSE+CGEN per line, wraps result in JS switch/trampoline.

Output format

The compiled JS uses a switch/case trampoline pattern:

var v={},pc=10,halt=false,out=[];
while(!halt){switch(pc){
  case 10: v['X']=1; pc=20; break;
  case 20: out.push(String(v['X'])); pc=30; break;
  case 30: halt=true; break;
}}

Supported keywords

ASIGN, PRINT, END, GOTO, IF/THEN/ELSE (with GOTO, ASIGN, PRINT actions), STRLP/NDLP, BREAK, IMPRT, CALSB, RTURN, REM, LOCAL, ONERR, WAIT, SYNC. Expressions support MOD, AND, OR, NOT, string concatenation (+), inline functions (MID, CHAR, STRLEN, INDEXOF, ARR, INT, ABS).

Example

10 IMPRT STRING
20 IMPRT CONVERT
30 ASIGN ML-PROG = "10 ASIGN X = 1|20 PRINT X|30 ASIGN X = X + 1|40 IF X <= 5 THEN GOTO 20|50 END"
40 CALSB 100-COMP
50 PRINT CO-OUT
60 END

CO-OUT will contain the complete JS switch/trampoline string.

The native CALSB COMPILE-Spark is the recommended path for most uses — it supports all keywords, multi-window programs, subroutines, and produces cleaner output. The Bootstrap COMP/CGEN/XFRM windows exist as a demonstration that Spark can compile Spark.
